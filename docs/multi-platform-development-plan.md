# 多平台商品运营中台开发计划

更新日期：2026-07-12

## 产品目标

将现有 Temu / 店小秘自动上品板块演进为多平台商品运营中台。系统只维护一份标准商品资料，通过平台适配器转换为 Temu、TikTok Shop、Shopee、Amazon 等平台刊登，并统一处理发布任务、失败恢复、规则校验和状态回传。

平台接入顺序：Temu -> TikTok Shop -> Shopee -> Amazon。

## 核心原则

- 保持现有 Temu 生产链路可用，采用渐进迁移，不整体重写。
- 区分销售平台、店铺账号和发布渠道。店小秘属于发布渠道，不等同于 Temu。
- 平台规则配置化、版本化，并记录站点、店铺模式、来源和验证时间。
- 官方 API 优先，其次稳定的 ERP 渠道，最后才是浏览器自动化和插件辅助。
- 高风险写操作必须经过目标页面、账号、规则和结果回查门禁。

## 里程碑

### M1：多平台后台框架与领域模型（1-2 周）

- 完成后台侧栏、顶部平台/店铺上下文和响应式布局。
- 建立销售平台、发布渠道、店铺账号、标准商品、平台刊登和发布任务模型。
- 保持现有 Temu 页面与接口行为不变。
- 输出旧模型到新模型的迁移映射。

验收：共享模型类型检查通过；现有 Temu 类型和流程无行为回归；后台能够明确显示当前平台和店铺。

### M2：标准商品中心（2-3 周）

- 建立标准 SPU/SKU、媒体、供应链、合规和多语言字段。
- 将 CSV、手工录入、1688 和店小秘采集归一到标准商品。
- 支持商品去重、版本记录和字段来源追踪。
- 新增商品列表、详情和平台刊登标签页。

验收：同一商品可以生成多个平台刊登草稿，且平台字段不污染标准商品。

### M3：通用发布编排与 Temu 迁移（2-3 周）

- 定义平台适配器能力契约。
- 将通用队列、重试、日志、截图、人工介入从店小秘命名中抽离。
- 把现有店小秘发布实现注册为 Temu 的首个发布渠道适配器。
- 提供兼容 API，逐步将 `/dianxiaomi/*` 迁移到通用资源路径。

验收：原 Temu 全流程继续通过；任务记录能同时表达平台、店铺、站点和发布渠道。

### M4：TikTok Shop 规则研究与草稿发布（3-5 周）

- 按目标站点和店铺模式建立官方规则矩阵。
- 研究类目、属性、标题、媒体、SKU、物流、资质和禁限售要求。
- 接入 TikTok Shop 官方开放接口；接口缺失能力再评估浏览器辅助。
- 完成类目映射、内容本地化、合规校验、定价和草稿创建。
- 第一版保留人工确认发布。

验收：真实账号完成小批量草稿创建；错误能够归类并从失败步骤恢复；规则均带来源和更新时间。

### M5：批量运营能力（3-4 周）

- 多店铺批量发布与发布模板。
- 库存、价格和刊登状态回传。
- 失败原因统计、账号健康和规则过期提醒。
- TikTok Shop 小批量发布达到稳定成功率后，逐步开放自动发布。

### M6：Shopee 与 Amazon（持续迭代）

- 复用平台适配器契约接入 Shopee 多站点。
- 在合规、品牌和类目治理成熟后接入 Amazon。
- 每个平台先完成草稿和人工确认，再基于实测证据提高自动化等级。

## 平台规则知识包

每个平台必须维护以下资料：

- 官方规则与接口文档
- 站点及跨境/本土店差异
- 类目、属性和资质矩阵
- 标题、图片、视频和详情要求
- SKU、价格、库存、重量尺寸和物流约束
- 禁限售、侵权、敏感词和审核驳回原因
- API 权限、调用频率和状态机
- 真实后台采样、失败案例和已验证经验

规则级别分为 `mandatory`、`recommended`、`observed` 和 `unverified`。只有官方强制规则和已实测规则可以作为自动阻断条件。
